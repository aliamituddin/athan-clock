446  /*
447  * `NcpFrameBuffer` Implementation
448  * -------------------------------
449  *
450  * `NcpFrameBuffer` internally stores a frame as a sequence of data segments. Each segment stores a portion of
451  * frame. The data segments are stored in the the main buffer `mBuffer`. `mBuffer` is utilized as a circular buffer.
452 
453  * The content of messages (which are added using `InFrameFeedMessage()`) are not directly copied in the `mBuffer`
454  * but instead they are enqueued in a message queue `mMessageQueue`.
455  *
456  * Every data segments starts with a header before the data portion. The header is 2 bytes long with the following
457  * format:
458  *
459  * Bit 0-13: Give the length of the data segment (max segment len is 2^14 = 16,384 bytes).
460  * Bit 14: Flag bit set to indicate that this segment has an associated `Message` (appended to its end).
461  * Bit 15: Flag bit set to indicate that this segment defines the start of a new frame.
462  *
463  * Bit 15 Bit 14 Bits: 0 - 13
464  * +--------------+--------------+--------------------------------------------------------+
465  * | New Frame | Has Message | Length of segment (excluding the header) |
466  * +--------------+--------------+--------------------------------------------------------+
467  *
468  * The header is encoded in big-endian (msb first) style.
469 
470  * Consider the following calls to create a frame:
471  *
472  * ncpBuffer.InFrameBegin();
473  * ncpBuffer.InFrameFeedData("Hello", 5);
474  * ncpBuffer.InFrameFeedData("There", 5);
475  * ncpBuffer.InFrameFeedMessage(*someMessage);
476  * ncpBuffer.InFrameFeedData("Bye", 3);
477  * ncpBuffer.InFrameEnd();
478  *
479  * This frame is stored as two segments:
480  *
481  * - Segment #1 contains "HelloThere" with a header value `0xC00A` which shows that this segment contains 10
482  * data bytes, and it starts a new frame, and also includes an appended message from the message queue.
483  *
484  * - Segment #2 contains "Bye" with a header value of `0x0003` showing length of 3 and no appended message.
485  *
486  * +-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+
487  * | C0 | 0A | 'H' | 'e' | 'l' | 'l' | 'o' | 'T' | 'h' | 'e' | 'r' | 'e' | 00 | 03 | 'B' | 'y' | 'e' |
488  * +-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+-----+
489  * \ / \ /
490  * Segment #1 Header Segment #2 Header
491  *
492  *
493  * `NcpFrameBuffer` uses the `mBuffer` as a circular/ring buffer. To support two frame priorities the buffer is
494  * divided into two high-priority and low-priority regions. The high priority frames are stored in buffer in
495  * backward direction while the low-priority frames use the buffer in forward direction. This model ensures the
496  * available buffer space is utilized efficiently between all frame types.
497  *
498  * mReadFrameStart[kPriorityLow]
499  * |
500  * | mWriteFrameStart[kPriorityLow]
501  * | |
502  * V Low Priority Frames V
503  * --------------+------------------------------+------------------------------+------------------
504  * ... | <-------- | --------> | ...
505  * --------------+------------------------------+------------------------------+------------------
506  * ^ High Priority Frames ^
507  * | |
508  * | mReadFrameStart[kPriorityHigh]
509  * |
510  * mWriteFrameStart[kPriorityHigh]
511  *
512  *
513  *
514  * When frames are removed, if possible, the `mReadFrameStart` and `mWriteFrameStart` pointers of the two priority
515  * levels are moved closer to avoid gaps.
516  *
517  * For an output frame (frame being read), NcpFrameBuffer maintains a `ReadState` along with a set of pointers
518  * into the buffer:
519  *
520  * mReadFrameStart[priority]: Start of the current/front frame.
521  * |
522  * | mReadSegmentHead: Start of the current segment.
523  * | |
524  * | | mReadPointer: Pointer to the next byte to read.
525  * | | |
526  * | | | mReadSegmentTail: End of the current segment.
527  * | | | |
528  * V V V V
529  * ---------+------------+--------------------------+------+----------------+-----------------------------------
530  * ... | Segment 1 | Segment 2 | ... | Last Segment | ... (possible) next frame
531  * ---------+------------+--------------------------+------+----------------+-----------------------------------
532  * \ | | /
533  * | \------------v-----------/ |
534  * | Current Segment |
535  * | |
536  * \---------------------------V-----------------------------/
537  * Current OutFrame (being read)
538  *
539  * Note that the diagram above shows the pointers for a low-priority frame (with pointers increasing in forward
540  * direction).
541  *
542  * The `ReadState` indicates the state of current output frame and its read offset (e.g., if read offset is in
543  * middle of a segment or if it is is middle of an appended message, or if we are done with entire frame).
544  *
545  * For an input frame (frame being written), the following pointers are maintained:
546  *
547  * mWriteFrameWrite[priority]: Start of the current/next frame being written.
548  * |
549  * | mWriteSegmentHead: Start of the current segment of the active input frame.
550  * | |
551  * | | mWriteSegmentTail: Pointer to the next byte to write.
552  * | | |
553  * | | |
554  * | | |
555  * V V V
556  * ------------------+------------------+-------------------------------------------------------------------
557  * Previous Frames | Segment 1 | Segment 2 : . . .
558  * ------------------+------------------+-------------------------------------------------------------------
559  * \ /
560  * | |
561  * \------------------V-----------------/
562  * Current InFrame (being written)
563  *
564  *
565  */
